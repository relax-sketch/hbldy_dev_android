## Context

当前仓库包含一个基础 Android Compose 应用、草原监测 `.zdb` 结构分析文档、一个可读取但业务表为空的 ZDB 样本，以及包含实际业务记录的 JSON 样本。只读核验表明，样本 ZDB 是 SQLite 数据库，含 56 张表和 `FS_DATACHECK_ITEM` 中 264 条启用规则；这些规则未携带可靠的提示性/强制性分类。

用户的现场文件结构为 Android 内部共享存储中的 `草原监测/数据/{项目目录}/{项目名}.zdb`。质检是反复流程：工作人员发现错误后在原采集系统修正，再回到 App 重新检查；App 不得修改 ZDB，只需呈现错误解释和实际命中值，并允许人工忽视仍然命中的问题。

## Goals / Non-Goals

**Goals:**

- 建立可安装 Android App 的后续实施设计，使用户能选择数据目录、发现样地并按单样地、区县或全域执行质检。
- 使用随 APK 发布的统一 JSON 规则集，将 ZDB 基础规则快照与首批 10 条附加 SQL 放在同一执行路径。
- 保证用户 `.zdb` 在扫描和检查过程中始终只读。
- 提供面向复检的结果查看和人工忽视状态存储，且忽视不改变 SQL 命中事实。
- 使检查引擎、规则配置和界面状态具有后续增加规则、历史对比或更多筛选的扩展边界。

**Non-Goals:**

- 本 change 工件阶段不修改源码、不生成规则资产、不开始实现。
- 首版不在 App 内编辑或导入规则，不导出检查报告。
- 首版不写回用户 ZDB，不保存业务数据副本，不提供完整检查历史对比。
- 首版不提供后台长时间执行、断点续检或通知栏检查任务。

## Decisions

### 1. 使用 APK 内置 JSON 规则集，不在运行时读取待检库规则

规则存放在后续实现的 `assets/rules/*.json` 中。首版规则集包括从已验证 ZDB 提取的 264 条启用规则快照，以及从草地规则表抽取并 SQL 化的 10 条附加规则。基础规则统一标记为强制性，附加规则保留自然语言规则表中的等级。

采用该方式是因为规则版本可随 APK 审核和测试，避免不同待检 ZDB 携带不同规则导致不可重复结果。替代方案是在运行时加载每个 ZDB 的 `FS_DATACHECK_ITEM`，但这会引入规则变异与解析兼容风险；使用 App 内 SQLite 规则库则在当前没有动态规则管理需求时增加了不必要的迁移成本。

### 2. 直接只读查询用户 ZDB，不复制到检查中间库

扫描与检查连接必须以只读方式打开用户 ZDB 并启用 SQLite 查询只读保护。SQL 规则仅允许单条 `SELECT` 或 `WITH ... SELECT`；人工标注写入 App 私有存储而不是业务库。

该方式比旧森林原型中“复制数据到检查结果库再执行”的做法更适合当前需求，因为草地首版不做跨批次快照，也避免复制多级关联表时造成数据遗漏或新结构适配负担。

### 3. 通过 `PlotRef` 统一单样地和批量执行

扫描阶段为每个样地生成包含来源 URI、原始 `YD_ID`、显示 ID、区县、项目摘要和可用关联标识的 `PlotRef`。检查引擎只接受 `List<PlotRef>`：单样地为单元素集合，区县和全域为多元素集合。

相同 `YD_ID` 仅在显示层追加 `-2`、`-3`，其身份仍由来源 URI 与原始 ID 确定。该选择可在不复制检查逻辑的前提下扩展到未来的多选或项目范围检查。

### 4. 每条规则必须限制到目标样地并返回可解释内容

规则配置必须定义 ID、来源、等级、目标表、标题、解释、所需表字段和查询 SQL。查询命中即生成问题，且查询结果必须包含定位字段和实际错误值。主表直接以 `YD_ID` 约束，子表按其 `YD_ID` 或 `MZGUID`/`XB_GLH` 关联链限制到目标样地。

规则所需表或字段缺失、或者无法安全限定到目标样地时，该规则对该样地记录为跳过，而不是脱离作用域查询整库。

### 5. 将忽视作为 App 私有标注，而非“通过”

结果状态分为当前命中的 `待处理` 与当前命中但由人工标注的 `已忽视`。忽视问题指纹由来源 ZDB URI、原始样地 ID、规则 ID 和记录定位值构成。复检仍命中相同指纹时，恢复绿色已忽视状态并排序到末尾；复检不再命中时，不显示在当前结果。

首版不保存完整历史快照，因此不单独展示“已修复”历史清单。该选择满足现场重复检查需求，同时不给 SQL 判定赋予错误语义。

### 6. 使用分层但精简的 Compose 工作流

后续界面按数据源选择、检查范围选择、检查进度、结果汇总和样地问题详情组织。范围页面包含默认全域的区县筛选、样地 ID 手工输入/列表选择和三种检查范围。批量结果以样地汇总作为入口，详情页复用单样地结果视图。

该结构比为每种范围建立独立流程更小，同时能够后续加入历史、备注、更多筛选和导出功能。

## Risks / Trade-offs

- [ZDB 版本变化导致内置规则失配] -> 规则执行前校验表和字段，将不兼容规则明确记为跳过；规则更新通过 APK 新版本发布。
- [264 条基础规则未具备可靠等级] -> 首版统一标记强制性并在后续规则维护工作中允许人工重分类。
- [基础规则原始条件未天然限定单一样地] -> 规则转换阶段为每条执行查询增加受控作用域，并以单样地/批量一致性测试阻止越界误报。
- [当前 ZDB 样本业务表为空，无法直接验证命中结果] -> 使用现有 JSON 业务样本构造测试 ZDB，并为首批附加规则配置命中与不命中用例。
- [全域批量执行时间可能在手机上较长] -> 首版显示前台进度并支持取消；后台任务与断点续检留作后续能力。
- [忽视状态在文件 URI 或定位值变化后无法关联] -> 指纹采用来源与定位值保证不误匹配；未来如需要跨导入迁移，再增加标注迁移策略。

## Migration Plan

1. 在后续实施中建立规则 JSON schema、规则提取/转换产物和自动验证测试。
2. 增加 ZDB 扫描、样地索引与只读检查引擎，并以构造测试库验证安全与作用域。
3. 增加 App 私有忽视标注存储与 Compose 工作流。
4. 生成 APK，在真实 Android 数据目录授权流程和项目目录结构下完成手工验收。
5. 若规则或结构发生不兼容，回滚到上一已验证 APK 版本；用户 ZDB 无迁移或回滚操作，因为 App 不修改该文件。

## Open Questions

无阻塞首版设计的问题。规则历史对比、忽视备注、结果导出和后台长任务均明确留作后续 change。
