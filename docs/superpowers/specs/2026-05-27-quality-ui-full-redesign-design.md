# 草原监测质检全页面高拟真重构设计

日期：2026-05-27

## 背景

当前项目是一个基于 Android Compose 的单 Activity 应用，围绕草原监测 ZDB 数据执行只读质检。现有实现已经具备完整业务流程：

1. 数据目录选择
2. 选择质检范围
3. 质检进行中
4. 质检结果
5. 样地详情

当前问题不在后端能力，而在前端表现：

- 页面视觉风格较接近默认 Material 组件，和目标稿差距明显
- 页面结构集中在 `QualityCheckApp.kt`，不利于大幅视觉重构
- 结果页、详情页、范围页的信息层级和空间分配较弱
- 还没有把平板和横竖屏适配作为正式约束纳入页面结构

用户提供了 `D:\software_dev\图片参考` 下的参考图，要求对所有主流程页面做高拟真复刻，尽量贴近参考图的配色、布局层级、卡片组织、底部操作区和视觉节奏，同时尽量不改后端和业务逻辑。

## 目标

1. 重构 5 个主流程页面的前端结构和视觉表现：
   - 页面入口
   - 选择质检范围
   - 质检进行中
   - 质检结果
   - 样地详情
2. 把空状态、错误态、取消态、禁用态纳入同一套视觉系统。
3. 保持现有业务语义：
   - 目录扫描与只读访问
   - 单样地/全量质检
   - 测试模式仅控制已通过规则显示
   - 忽略/取消忽略
   - 结果导航与返回层级
4. 纳入手机和平板、横屏和竖屏适配。
5. 将 UI 层从当前的大文件实现拆分成可维护的页面层和通用视觉组件层。
6. 在实施阶段，重构每个页面前必须重新读取该页面对应的参考图，不凭记忆直接改 UI。

## 非目标

1. 不修改 ZDB 读取和规则执行语义。
2. 不调整 `QualityCheckEngine`、只读数据库访问、规则分类和忽略逻辑的核心行为。
3. 不引入深色模式和 Android 动态取色适配。
4. 不把本次改造升级为一套过度抽象的跨项目设计系统平台。
5. 不强行让所有横屏和平板页面一比一照搬手机竖屏稿比例；宽屏下优先保证层级、可读性和操作可达性。

## 设计原则

### 1. 高拟真复刻，但不牺牲真实功能

本次不是简单换皮，而是按参考图重组信息层级、卡片区块、按钮位置和视觉节奏。但所有视觉重构都必须服从现有业务语义，不能为了像图而制造误导性的假状态。

### 2. 后端不动，前端中等规模重构

保留 `QualityCheckViewModel`、`QualityCheckUiState` 主体、现有导航语义和质检引擎。改造重点放在 Compose 页面结构、主题 token、页面 section 组织和公共 UI 组件。

### 3. 固定浅色主题

目标稿是固定浅色视觉系统，因此不保留深色模式和动态色。统一用固定设计 token 控制背景、阴影、描边、状态色、主按钮渐变和圆角。

### 4. 手机优先，但从一开始支持自适应

手机竖屏是最接近参考图的主形态；手机横屏、平板竖屏、平板横屏从同一套布局规则派生，不后补。

### 5. 文案以示意图优先，功能语义冲突时停下来确认

默认贴近示意图文案和标签风格，但如果图中写法和现有功能语义冲突，不直接强改，先向用户确认。

## 当前代码结构与边界

现有关键边界如下：

- `QualityCheckApp.kt`：承载路由、顶部栏和 5 个页面的 UI
- `QualityCheckViewModel.kt`：维护目录、扫描结果、过滤状态、质检进度、结果和详情
- `QualityCheckUiState.kt`：页面状态模型
- `IssueReviewService`、`QualityCheckEngine`：结果执行与复核
- `DetailResultFilters`：详情页按状态和业务表分组过滤

这些边界说明：

- UI 可以大改，但业务数据入口已经稳定
- 当前详情页已具备状态筛选和表分组筛选能力，不能丢，只能降权重做视觉表达
- 单样地完成后进详情、全量完成后进汇总已经符合目标语义，应保留

## 总体方案

采用“页面拆分 + 固定设计 token + 保留现有状态流”的方案。

### 保留不动的层

- `QualityCheckViewModel`
- `QualityCheckUiState` 的业务字段和核心 screen 语义
- 扫描、索引、质检执行、取消、忽略、结果复核流程
- `screenAfterCompletedCheck` 和 `screenAfterDetailBack` 这类导航规则

### 重构的层

- 页面容器和布局结构
- 顶部栏和底部操作区
- 统计块、卡片、标签、按钮、空态/错误态呈现方式
- 主题颜色、阴影、圆角、间距
- 页面内组件拆分

### 不采用的方案

- 不走“只在现有单文件中改样式”的轻量改法，无法满足高拟真复刻
- 不走“完整框架级响应式设计系统平台”的重型方案，超出当前项目收益

## 视觉系统

### 色彩

主基调：

- 页面背景：暖白偏粉的浅色背景
- 卡片背景：纯白或极浅暖白
- 主绿色：用于 CTA、成功态、选中边框、强调数字
- 辅助灰蓝：用于说明文案、次级文本和禁用态

状态色语义：

- 强制性：红
- 提示性：橙
- 已忽略：灰
- 已跳过规则：紫
- 已通过规则：绿
- 已执行规则：蓝

规则：

- 状态色优先以标签、数字、局部图标表现，不大面积铺满整张卡片
- 功能信息不能只靠颜色表达，必须带文字或图标

### 圆角与阴影

- 页面主卡片统一大圆角
- 顶部返回按钮使用独立白色圆角按钮
- 阴影轻量，不使用重 Material elevation 观感
- 统一描边体系，避免局部随机边框粗细

### 按钮

- 主按钮：绿色渐变、整宽、高度统一、白字
- 次按钮：白底、细描边、深色文字
- 危险按钮：白底或浅底，红字，不和主按钮抢层级

### 标签与状态胶囊

- 区县、统计项、状态数量统一使用胶囊风格
- 选中、禁用、弱提示都有对应轻量视觉层级

## 补充状态设计

所有补充状态进入同一视觉系统：

- 空状态
  - 无可读 ZDB
  - 没有匹配样地
  - 当前筛选无结果
  - 暂无质检结果
- 错误态
  - 扫描失败
  - 质检失败
  - 未选择样地
- 取消态
  - 本次质检已取消，但保留已完成结果
- 禁用态
  - 全量模式下的筛选控件和列表禁用

规则：

- 错误信息不悬空显示为普通红字，应进入轻提示卡或内嵌状态块
- 空状态不只是一行文字，要有明显占位和恢复路径

## 自适应设计

### 尺寸分层

定义三个布局等级：

- `Compact`：手机竖屏
- `Medium`：手机横屏、小平板竖屏
- `Expanded`：平板横屏、大宽度窗口

### 各页面策略

#### 入口页

- Compact：单列堆叠
- Medium/Expanded：目录卡与统计卡可重排为更舒展布局

#### 范围页

- Compact：单列，尽量贴近参考图
- Medium：筛选区更宽，列表高度更稳定
- Expanded：左筛选、右列表双栏

#### 进行中页

- Compact：纵向信息流
- Expanded：进度主卡在左，最近完成列表在右

#### 结果页

- Compact：统计卡 + 列表纵向
- Expanded：统计卡更宽，列表仍维持单列阅读，不做多列瀑布

#### 详情页

- Compact：单列分区
- Medium/Expanded：摘要区更舒展，筛选条和问题卡片增加横向留白

## 状态流与数据流

### 路由

维持当前路由规则：

- `SOURCE -> SCOPE -> PROGRESS -> SUMMARY/DETAIL`
- 单样地完成后进入详情
- 全量完成后进入汇总
- 详情根据来源返回范围页或汇总页

### 页面事件

页面只消费 view model 暴露的事件，不在页面中自行维护业务状态：

- 目录选择、重新扫描
- 更新区县
- 更新样地搜索
- 选择样地
- 切换全量/测试
- 启动质检、取消质检
- 再次质检
- 进入详情
- 忽略/取消忽略
- 设置详情筛选

### 图片驱动实施规则

实施阶段严格遵守：

- 每开始重构一个页面前，都先重新读取该页面对应参考图
- 不凭当前对话截图记忆直接编写页面
- 如果参考图和功能语义冲突，以功能为准并先向用户确认

## 风险与处理

### 1. 参考图中存在装饰性状态点或样例数据

风险：

- 某些视觉元素未必有真实业务语义

处理：

- 只能将其作为弱装饰或辅助状态，不绑定虚假业务含义

### 2. 详情页筛选会破坏复刻感

风险：

- 当前详情页已有筛选能力，而参考图更偏内容审查页

处理：

- 保留筛选功能，但视觉降权为轻量胶囊条；不把工具控件扩展成大块布局

### 3. 宽屏强行照抄手机稿会导致横向浪费

风险：

- 平板和横屏下一比一照搬手机比例会显得空和长

处理：

- 保持视觉风格一致，但宽屏使用双栏和 max width 布局，不追求完全同形

### 4. 当前工作区已存在未提交改动

风险：

- 本次设计文档不应污染用户已有修改

处理：

- 设计文档单独新增、单独提交，不回退任何用户代码

## 结论

本次改造应视为一次“前端结构重组 + 高拟真视觉重建”，而不是单纯样式替换。最优路径是在不触碰后端执行链路的前提下，围绕现有 screen 和 state 流，拆出稳定的 Compose 页面层与通用视觉组件层，并把参考图驱动的复刻过程和自适应布局一起纳入实施约束。
