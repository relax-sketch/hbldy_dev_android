## 规则软禁用

规则资源仍以现有 `rule-set.json` 为来源。本次将 70 条 JM 相关规则作为“暂时禁用”处理，范围由目标表、必填表或 SQL 中包含 JM 相关表名判定。已确认数量为 70 条，按目标表分布如下：

| 目标表 | 数量 |
| --- | ---: |
| `YD_JM_PT` | 42 |
| `YF_JMCC_TB` | 7 |
| `ZWDCB_JMCCYF_TB` | 9 |
| `YF_JMGDCG_TB` | 4 |
| `ZWDCB_JMGDCG_TB` | 8 |

禁用规则不参与执行，也不进入强制性、提示性、跳过、忽略、通过等任何统计。实现上优先使用显式禁用标记或禁用 ID 列表，让后续恢复可以通过撤销标记完成，而不是删除规则内容。

## 样地详情页状态筛选

详情页顶部使用 5 个小色块统计块：

| 色块 | 含义 | 点击后筛选 |
| --- | --- | --- |
| 全部 | 当前样地当前表分组下所有可展示规则结果 | 展示全部状态 |
| 未通过 | 未忽略的强制性问题和未忽略的提示性问题 | 只展示未通过问题 |
| 强制性 | 未忽略的强制性问题 | 只展示强制性问题 |
| 提示性 | 未忽略的提示性问题 | 只展示提示性问题 |
| 已忽略 | 用户已忽略的问题 | 只展示已忽略问题 |

`未通过` 不包含跳过规则，也不包含已忽略规则。跳过规则不单独做顶部色块，但在选择“全部”并符合表分组时仍可展示。

统计数字跟随当前表分组变化。点击某个色块后，列表只显示对应状态；再次点击同一状态可以保持选中，不需要做取消选中逻辑，“全部”用于恢复全量状态。

## 业务表分组筛选

详情页增加一个按业务表分组的筛选控件，选项为：

1. 全部
2. 样地表
3. 样线表
4. 测产样方
5. 观测样方
6. 高大草灌

业务分组面向用户展示，不直接暴露数据库表名。分组筛选对未通过、已忽略、跳过、通过结果都生效。植物调查表不作为单独选项展示，而是并入对应业务分组：

| 业务分组 | 包含内容 |
| --- | --- |
| 样地表 | 样地基础表相关规则 |
| 样线表 | 样线相关规则 |
| 测产样方 | 测产样方及其植物调查表 |
| 观测样方 | 观测样方及其植物调查表 |
| 高大草灌 | 高大草灌及其植物调查表 |

实现时需要为规则结果提供可稳定推导的业务分组。优先从规则目标表、必填表或 SQL 涉及表名映射得到分组；不能识别的结果归入“全部”可见，但不进入具体业务分组选项，避免误归类。

## 结果数据流

质检执行阶段继续产出问题、跳过、通过等结果。禁用规则在执行前过滤。执行后的结果增加或可推导出两个维度：

1. 状态筛选维度：全部、未通过、强制性、提示性、已忽略。
2. 业务表分组维度：全部、样地表、样线表、测产样方、观测样方、高大草灌。

详情页 UI 状态保存当前选中的状态筛选和业务分组筛选。列表渲染、顶部统计和标题下方汇总均从同一份过滤模型计算，避免统计和列表不一致。

## 测试 ZDB 错误注入

只处理本地测试副本：

`D:\software_dev\test\综合监测-枣阳市-20260527103022.zdb`

注入策略为稳定、可复现、覆盖关键业务分组：

1. 先读取 schema 和样地列表，确认表结构和主键/关联字段。
2. 选择多个样地，按样地表、样线表、测产样方、观测样方、高大草灌分别制造少量明确错误。
3. 错误类型优先选择规则容易命中的字段，例如必填缺失、数值范围异常、关联缺失或逻辑矛盾。
4. 修改前记录文件 hash，修改后再次记录 hash。
5. 输出注入摘要：影响表、影响样地、修改字段、预期触发的规则类型。

该文件允许直接覆盖。不会修改手机原始路径下的 ZDB。

## 风险与处理

1. 规则到业务分组的映射可能存在遗漏。处理方式是先集中维护映射，并把无法识别的规则留在“全部”中，避免错误归类。
2. 禁用 JM 规则可能影响总统计数量。处理方式是明确禁用规则完全不参与结果模型，测试中校验统计口径。
3. ZDB 表结构可能与预期不完全一致。处理方式是先读取 schema，再按实际字段生成注入语句，不硬编码不可验证的字段。
4. 当前工作区已有未提交改动。处理方式是提交设计文档时只暂存该文档，后续实现继续在现有改动基础上工作，不回退用户改动。
