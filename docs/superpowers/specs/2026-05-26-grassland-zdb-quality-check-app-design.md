# 草原监测 ZDB 质检 Android App 首版设计

## 1. 目标

构建一个可安装的 Android APK，用于在手机端对草原监测项目的 `.zdb` 数据进行只读质量检查。App 允许人工选择手机内部共享存储中的 `草原监测/数据` 目录，扫描其项目子目录中的 `.zdb` 文件，按单个样地、指定区县或全部样地执行内置 SQL 规则，并将命中的具体错误值和规则解释展示给用户，辅助用户回到原业务系统修正数据后再次质检。

首版的核心目标是质检后端链路完整可验证，界面只实现完成现场使用所必需的流程，并为后续扩展保留清晰边界。

## 2. 已验证的输入事实

- 测试文件 `test/综合监测-江岸区-20260526102825.zdb` 是可正常打开的 SQLite 3 数据库，文件大小为 649216 字节，`PRAGMA quick_check` 返回 `ok`。
- 当前 `.zdb` 样本包含 56 张表，其中 `FS_DATACHECK_ITEM` 有 280 行，启用规则有 264 条。
- 当前 `.zdb` 样本的业务数据表为空，适合作为结构和规则来源；`test/*.json` 提供了实际草地业务记录样本，可用于构造检查测试库。
- 264 条启用基础规则没有可靠的提示性/强制性标签，首版统一标记为 `强制性`。
- `草地样地调查质检规则表.xlsx` 提供自然语言规则及其等级；首版仅抽取其中 10 条可 SQL 化规则作为附加规则验证集，保留其原始 `强制` 或 `提示` 属性。
- `.zdb` 的 SQLite 编码声明为 `UTF-8`。实现不得预设所有字段必须按 GBK 转换；如后续出现具体乱码字段，再针对证据处理。

## 3. 首版范围

### 3.1 包含功能

- 使用 Android 系统目录选择器选择 `草原监测/数据` 目录并保留读取授权。
- 递归扫描目录下项目子文件夹中的所有 `.zdb` 文件。
- 从有效 `.zdb` 建立样地索引，并按区县筛选、手动输入样地 ID 或列表选择样地。
- 支持三种检查范围：
  - 单个样地；
  - 当前选中区县的全部样地；
  - 已扫描数据源中的全部样地。
- 内置统一 JSON 规则配置：
  - 从样本 `.zdb` 提取的 264 条启用基础规则快照；
  - 从草地规则表中 SQL 化的首批 10 条附加测试规则。
- 对所选样地及关联样线、样方、植物调查记录进行只读 SQL 检查。
- 展示强制性问题、提示性问题、跳过规则及每条命中问题的实际字段值和解释。
- 支持人工将问题标注为 `已忽视`，以及取消忽视。
- 支持按当前检查范围重新质检；仍然命中的已忽视问题继续保持已忽视状态并置于问题列表底部。
- 批量检查时展示进度与按样地汇总的结果入口。

### 3.2 不包含功能

- 不修改、修复或写回任何用户 `.zdb` 文件。
- 不在首版提供 App 内规则编辑、外部规则导入或规则导出。
- 不导出质检报告。
- 不保存完整检查批次历史，也不展示新增问题、已修复问题的跨批次对比。
- 不执行后台长期批量检查、断点续检或通知栏任务。

## 4. 关键产品决策

### 4.1 规则以内置快照为准

App 不在运行时使用待检 `.zdb` 中的规则表作为规则来源。基础规则从已验证样本中提取后随 APK 发布，附加规则与基础规则存放在同一套 JSON 配置中。遇到新版本数据结构或规则变更时，通过升级 APK 更新规则快照。

该选择降低了运行时规则变异、规则 JSON 解析差异和外部库规则被调整造成的风险，同时保持待检 `.zdb` 完全只读。

### 4.2 问题忽视不是检查通过

规则 SQL 命中即表示数据仍不满足该规则。人工 `已忽视` 只改变处理状态，不改变规则判定：

- `待处理`：SQL 当前命中，未被人工忽视。
- `已忽视`：SQL 当前仍命中，但人工选择暂不处理；在界面中以绿色状态显示，并排序到末尾。
- SQL 复检后不再命中的问题不进入当前问题列表。首版不提供历史已修复视图。

### 4.3 单样地与批量检查复用同一引擎

检查引擎以 `List<PlotRef>` 作为输入。单样地检查只传入一个元素，区县和全域检查传入多个元素。规则执行与结果构建不因检查范围产生第二套实现。

## 5. 体系结构

Android 工程保持单 App 模块，使用 Jetpack Compose 构建基础界面。业务边界建议如下。

### 5.1 `ZdbScanner`

职责：

- 接收系统目录选择器返回的目录 URI。
- 递归遍历后缀为 `.zdb` 的文件。
- 忽略并报告 0 字节文件、无法打开文件、非 SQLite 文件或不支持的结构。
- 以只读连接查询支持的样地主表中的样地索引信息。

输出模型：

```text
PlotRef
  sourceUri          来源 ZDB URI
  projectName        项目子目录/文件名摘要
  rawPlotId          数据库中的原始 YD_ID
  displayPlotId      UI 展示编号，重号时追加 -2、-3
  countyCode         县代码
  countyLabel        县展示信息
  plotTable          样地主表类型
  parentGuid         主表 MZGUID（可用时）
```

重复 `YD_ID` 不修改原数据，仅在 `displayPlotId` 中追加后缀，并依靠 `sourceUri + rawPlotId` 保持身份准确。

### 5.2 `RuleRepository`

职责：

- 从 APK `assets/rules/*.json` 加载统一规则集。
- 暴露规则版本、规则数量和规则明细。
- 在加载阶段执行规则配置静态校验。

首版规则集合：

- `264` 条基础规则：来源为 `.zdb` 自带启用规则的快照，等级统一设置为 `MANDATORY`。
- `10` 条附加规则：来源为草地自然语言质检规则，选择能够映射到当前表结构、能够用测试数据验证的规则；等级保留为 `MANDATORY` 或 `ADVISORY`。

规则 JSON 最小结构：

```json
{
  "id": "YD_TRCY_001",
  "source": "BASE_SNAPSHOT",
  "severity": "MANDATORY",
  "targetTable": "YD_TRCY_PT",
  "title": "调查人员为空",
  "explanation": "已调查样地必须填写调查人员。",
  "requiredTables": ["YD_TRCY_PT"],
  "requiredFields": ["YD_ID", "DC_RY", "XIAN", "XIANG", "CUN"],
  "sql": "SELECT XIAN, XIANG, CUN, YD_ID, DC_RY AS actualValue FROM YD_TRCY_PT WHERE YD_ID = :ydId AND (DC_RY IS NULL OR TRIM(DC_RY) = '')"
}
```

所有规则 SQL 必须返回界面需要展示的定位内容和实际错误值。

### 5.3 `QualityCheckEngine`

职责：

- 接收待检查的 `PlotRef` 集合和内置规则集合。
- 对每个来源数据库建立只读连接，并开启 SQLite 只读保护。
- 对规则执行结构兼容检查。
- 参数化执行规则并生成问题结果。
- 在批量场景报告进度和允许取消当前检查任务。

规则作用域：

- 样地主表使用原始 `YD_ID` 约束。
- 一级子表优先使用其 `YD_ID`；需要关联时使用主表 `MZGUID` 与子表 `XB_GLH` 的链路。
- 植物调查等二级子表通过所属样方的关联字段限制到选定样地。
- 规则无法在当前结构上定位到选定样地时，不得无范围地扫描并误报其他样地；应记录为跳过或由转换规则补足作用域。

输出模型：

```text
CheckIssue
  fingerprint        问题稳定指纹
  plotRef            对应样地与来源
  ruleId             规则 ID
  severity           强制性/提示性
  title              错误标题
  explanation        规则解释
  tableName          命中表
  locationValues     样地/样线/样方等定位值
  actualValues       命中的实际错误值
  ignored            是否人工忽视

SkippedRule
  plotRef
  ruleId
  reason             表缺失、字段缺失或结构无法定位等原因
```

### 5.4 `IssueAnnotationStore`

职责：

- 在 App 私有本地数据库中保存人工忽视标注。
- 按稳定问题指纹恢复复检后的忽视状态。
- 支持忽视与取消忽视操作。

问题指纹至少包含：

```text
来源 ZDB URI + 原始 YD_ID + 规则 ID + 记录定位值
```

本地数据库只保存标注元数据，不保存待检 `.zdb` 的业务数据副本。

### 5.5 `CheckViewModel`

职责：

- 协调目录授权、扫描、筛选、检查范围、执行进度和展示结果。
- 将后端模型映射为 Compose 界面状态。
- 在配置变化或页面返回时保存必要的瞬时筛选状态。

## 6. 用户流程与界面

界面遵循 Material 3 基础组件与可访问触控尺寸，首版聚焦操作闭环。

### 6.1 数据源页

- 选择数据目录按钮，进入 Android 系统目录选择器。
- 显示所选目录摘要，例如 `.../草原监测/数据/`。
- 扫描后显示项目子目录数、有效 `.zdb` 数和不可读取文件数。
- 提供重新扫描入口。

### 6.2 检查范围页

- 区县筛选控件，默认 `全部区县`。
- 样地 ID 输入框，支持手动输入过滤。
- 样地列表/下拉选择，显示样地号、区县和来源项目摘要。
- 检查范围选项：
  - 选中单个样地；
  - 当前区县全部样地；
  - 全部样地。
- 开始检查前显示选中范围、样地数量和规则版本摘要。

### 6.3 检查进度页或进度状态

- 批量执行显示 `正在检查 n / total`。
- 支持取消本次执行。
- 单样地可直接以加载状态进入结果页，无需独立复杂页面。

### 6.4 结果汇总页

单样地检查直接显示该样地的问题汇总。区县或全域批量检查首先显示按样地的汇总列表。

顶部统计：

- 已检查样地数；
- 存在待处理强制性问题的样地数；
- 待处理强制性问题总数；
- 待处理提示性问题总数；
- 已忽视问题总数；
- 已执行规则数与已跳过规则数。

批量样地汇总列表默认按待处理问题数量降序排列，点击样地进入详情页。

### 6.5 样地问题详情页

- 待处理问题显示在上方，强制性在提示性之前。
- 已忽视问题位于列表底部，以绿色状态条或绿色卡片区分。
- 问题详情展示规则标题、规则解释、检查表、定位值和实际命中值。
- 每项提供 `忽视` 或 `取消忽视` 操作。
- 支持对当前样地或当前原检查范围重新质检。
- 跳过规则可通过单独折叠区域查看跳过原因。

## 7. 安全与错误处理

### 7.1 SQLite 只读约束

- 对用户 `.zdb` 使用只读打开模式。
- 每个连接执行 `PRAGMA query_only=ON`。
- 不创建临时表、不写回用户库、不在用户库执行规则存储或结果存储。
- 忽视标注仅写入 App 私有数据库。

### 7.2 SQL 静态限制

规则加载时仅接受单条 `SELECT` 或 `WITH ... SELECT` 查询。拒绝包含写入或结构操作能力的语句，包括但不限于：

- `INSERT`、`UPDATE`、`DELETE`、`REPLACE`；
- `CREATE`、`DROP`、`ALTER`；
- `ATTACH`、`DETACH`；
- `PRAGMA`；
- 多语句执行。

规则 SQL 由 APK 构建期固定提供，不接受用户输入 SQL。

### 7.3 数据与结构异常

- 文件为 0 字节、损坏或不是 SQLite：扫描时记录为无效文件，不阻塞其余文件。
- 必需主表缺失：该文件不加入可检查样地索引，并显示结构不支持。
- 单条规则表或字段缺失：记为跳过并提供原因，其余规则继续运行。
- 重号样地：以来源文件区分并提供 UI 后缀，不合并、不覆盖。
- 已忽视问题复检仍命中：保留忽视状态；不再命中：从当前结果消失。

## 8. 测试与验收

### 8.1 规则与数据准备

- 提取当前样本 `.zdb` 的 264 条启用基础规则并生成统一 JSON。
- 从草地规则表选择 10 条与当前表结构匹配、适合验证的规则，转换为带解释和定位值输出的 SQL。
- 根据 `test/*.json` 构造含业务记录的测试 `.zdb`，避免仅依赖业务表为空的结构样本。

### 8.2 后端测试

- 规则 JSON 能被解析，并通过 SQL 只读静态限制检查。
- 每条附加规则至少提供一组命中记录和一组通过记录。
- 单样地、区县批量和全域批量执行，对同一具体样地产生一致的问题集合。
- 子表和二级关联表规则只返回所选样地的数据。
- 结构缺失规则被跳过且不导致整个检查任务失败。
- 重号样地能够分别检查正确来源数据库。

### 8.3 标注与复检测试

- 标注忽视后，问题移入底部绿色区域。
- 取消忽视后，问题回到待处理区域。
- 未修改源数据而重复质检时，相同问题继续保持已忽视。
- 测试数据修正后重新质检，已不命中的问题不再展示。

### 8.4 只读证明与设备验证

- 在检查前后比较待检 `.zdb` 文件摘要，确认检查流程未修改文件。
- 生成可安装 APK。
- 在真实 Android 文件选择流程中选择 `内部共享存储空间/草原监测/数据`，验证项目子目录扫描、样地筛选、三种检查范围、问题查看和忽视复检流程。

## 9. 后续扩展点

当前设计预留但首版不实现：

- 检查批次快照，以及新增、已修复、持续存在问题对比；
- 忽视原因和人工备注；
- 按表类型、规则标签和更多行政范围过滤；
- 规则版本详情与规则升级说明；
- 导出或分享结果摘要；
- 规则维护工具，以及对基础规则等级进行人工校订；
- 后台长时间批量检查与断点续检。

## 10. 实施边界

本设计批准阶段仅形成规范与后续实施任务，不修改 App 源码、不生成实际规则资产、不开始 Android 功能开发。后续实现必须依据已批准的设计和独立实施计划开展。
